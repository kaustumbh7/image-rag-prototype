Prototype walkthrough

Image RAG.

Text-to-image retrieval over ~120,000 COCO images, with grounded answers from Claude vision on top.

The problem

Find images by description.
Then ask questions about them.

  • Given a corpus of ~120,000 images and a natural-language query, return the best-matching ones.
  • Optionally hand those retrieved images to a vision model so it can answer a question grounded in what's actually there.
  • RAG, but with images as the retrieval target instead of text chunks.
Headline query

The kind of question the system is built for.

Give me 5 images with men wearing a hat
  • Text → CLIP text encoder → 512-d vector.
  • Vector → FAISS nearest-neighbour over all pre-embedded images.
  • Top-5 paths → thumbnails in the UI.
  • Optionally → Claude vision writes a grounded answer that references each image by index.
Where it works well

Subject + one or two attributes + setting.

The sweet spot — concrete nouns, simple verbs, common scenes.

Object + scene
a red double-decker bus on a city street
Concrete nouns and common scenes — squarely in CLIP's wheelhouse.
Activity
a woman walking
Activity verbs are everywhere in CLIP's training captions: walking, running, riding, playing.
Style or mood
a black and white photo of a person riding a bicycle at night
Stylistic descriptors embed cleanly — they're common alt-text language.
Where it struggles

Known CLIP failure modes.

Not bugs in the prototype — inherent to how embedding models represent meaning.

Counting
exactly three apples on a wooden table
No reliable notion of cardinality — returns apple-and-table-shaped images regardless of count.
Spatial / binding
a cat sitting to the left of a dog
CLIP encodes a bag of concepts — left/right and attribute-to-object binding are lost.
Negation
a kitchen without a microwave
"Without" embeds roughly like "with". CLIP cannot represent negation.

Build the index once. Query it forever.

Tech stack · components

What's actually under the hood.

Dataset

COCO 2017

~120k images. Diverse, well-known, free. Swap in DeepFashion or Unsplash Lite for domain-specific demos.

Embedding model

OpenCLIP ViT-B/32

LAION-2B pretrained. Small enough for MPS; queries are instant. ViT-L/14 or SigLIP for harder cases.

Vector index

FAISS IndexFlatIP

One matrix multiply, < 10 ms. IVF / HNSW only pay off above millions of vectors.

Normalization

L2 at write time

Vectors normalized at write time, so inner product equals cosine similarity.

RAG layer

Claude Sonnet 4.6

Vision. Images resized to ≤ 1024 px, base64-encoded, system prompt forces grounded answers.

UI

Streamlit

Model + index cached across reruns. Only the first query pays the load cost.

Evaluation

How would I measure if this is actually good?

  • Spot-checks. Representative queries across categories — fast, qualitative, catches regressions.
  • Recall@k on COCO captions. For each ground-truth caption, does its image land in the top-k? ViT-B/32 baseline ≈ 30–40% R@1, 60–70% R@5.
  • Per-category drill-down. Break recall by query type — counting, spatial, compositional, colour binding, negation — to map where the model fails.
  • RAG quality. Rubric-based human review on ~50 (query, retrieved images, answer) tuples. Then scale with LLM-as-judge.
  • Latency & cost. Track query time, embed throughput, $/RAG call — the constraints that bite first at scale.
Wrap-up

That's the system.

CLIP gives us a shared text-image space. FAISS gives us fast nearest-neighbour search over it. Claude gives us reasoning over the retrieved subset.

  • Next: stronger embedder (SigLIP / ViT-L/14) for compositional cases.
  • Next: cross-encoder re-ranker on top-50 to clean up top-5.
  • Next: metadata filtering (Chroma / Qdrant) for category and attribute pre-filters.
